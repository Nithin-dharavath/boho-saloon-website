     Fix: Navbar "Services" link not clickable / not navigating

     Context

     The user reports that the Services item in the Boho Bloom navbar cannot be
     clicked and does not navigate to the services page. The project already
     contains the destination page (templates/services.html), the data module
     (app/services_data.py), the SEO entry (SERVICES_SEO in app/seo.py), and a
     /services route (main.py:89‚Äď95). So the page itself can be reached by
     typing /services in the URL ‚ÄĒ the problem is the navbar link itself.

     After reading the current state of the repo, the actual cause is
     duplication + a missing state contract, not a missing route:

     - templates/index.html:91‚Äď117 contains an inline <nav class="navbar"> for
     the home page.
     - templates/services.html:13‚Äď39 also contains its own inline <nav 
     class="navbar"> because templates/base.html does not define a navbar block.
     - The two navbars are not the same markup: services.html adds
     class="navbar-link-active" to the Services link and adds an extra <li><a 
     href="/login" class="navbar-signin">Sign In</a></li> driven by {% if user 
     and user.id %}.
     - On the home page, the "Services" link itself is a normal <a 
     href="/services"> and is clickable. The route works.

     The real failure modes the user is most likely seeing are:

     1. Mobile menu: the desktop navbar is hidden on screens ‚Č§820px
     (style.css:947‚Äď965) behind opacity:0; pointer-events:none until
     .navbar-toggle is tapped. If the user has not tapped the hamburger, the
     link looks present but cannot be clicked ‚ÄĒ a classic "feels broken" UX bug.
     2. On the /services page, the inline navbar does not use url_for(...) for 
     assets (services.html:16 uses url_for, good) but does hardcode /static/... 
     and /login, /auth/me which is fine. However, the active link styling and 
     link semantics differ between the two pages, leading to a confusing visual
     where neither page looks "current" properly ‚ÄĒ and a user might think the
     link is dead.
     3. Cursor-glow overlay (script.js:75‚Äď89) sits at z-index: 9999 with
     pointer-events:none ‚ÄĒ verified safe, not a click blocker.
     4. No clickable bug on desktop ‚ÄĒ the link is genuinely clickable. If the
     user is on desktop and reports "cannot click", the most likely explanation
     is that the server isn't running, the route returns 500 due to a missing 
     services_data import, or the page renders but the user expected the home 
     navbar to be reused and so doesn't see the new /services link working in 
     the same place. The services_data import in main.py:15 is already in place,
     and the route already passes services_data=SERVICES to the template, so
     the page itself works.

     The fix below addresses all three: makes the mobile menu actually usable,
     deduplicates the navbar into a single Jinja partial in base.html so both
     pages render the same control, and adds a small active-state helper.
